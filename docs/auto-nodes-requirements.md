# kg-index Requirements

## Goal

Add a separate local binary, `kg-index`, that scans filesystem trees, parses files, and materializes a graph of directories, files, documents, and selected code symbols.

The main `kg` CLI stays thin and delegates indexing work to `kg-index`.

## Current KG State

### Node types currently enforced by validation

`src/validate.rs` and `docs/node-and-edge-fields-reference.md` currently treat these as the core supported types:

- `Concept`
- `Process`
- `DataStore`
- `Interface`
- `Rule`
- `Feature`
- `Decision`
- `Convention`
- `Note`
- `Bug`

There are also legacy single-letter node tokens in validation/encoding:

- `D`
- `F`

### Types present in serialization/encoding but not in the main validation docs

`src/graph.rs` can encode/decode additional types:

- `Attribute`
- `Entity`
- `OpenQuestion`
- `Claim`
- `Insight`
- `Reference`
- `Term`
- `Status`
- `Doubt`

These exist in the graph layer, but they are not part of the documented canonical ontology and are not reflected in the main schema docs.

### Important mismatch

- `is_valid_node_type()` accepts custom ASCII tokens.
- Edge type rules in `validate.rs` only apply when both nodes are in `VALID_TYPES`.
- So custom node types are currently allowed, but they bypass most semantic edge validation.

### Current properties

`NodeProperties` and `EdgeProperties` are generic and already cover:

- description
- domain_area
- provenance
- confidence
- created_at
- importance
- key_facts / alias
- valid_from / valid_to
- edge detail
- feedback metadata

What is missing for indexing:

- file path / relative path
- file hash
- mime type
- language
- extension
- byte size
- modified time
- line/column spans
- parser status / parse errors
- section hierarchy metadata

### Existing partial support

- `import_markdown.rs` can import Markdown frontmatter and notes.
- There is no structural parser for repository trees, document sections, or code symbols yet.

## `kg-index` Requirements

### Scope

1. Scan a base directory recursively.
2. Allow optional include filter, e.g. regex or glob.
3. Allow optional `scan_ignore_unknown` so unknown file types stay out of the graph by default.
4. Ignore common junk by default:
   - `.git`
   - `node_modules`
   - `target`
   - build artifacts
   - binary files
5. Parse many file types, with best-effort fallbacks.
6. Build a deterministic graph.
7. Support incremental reindexing.

### Output

`kg-index` should emit graph updates that can be stored by the existing KG backend, not a separate source of truth.

Recommended outputs:

1. graph nodes and edges
2. per-file metadata
3. parser warnings/errors
4. stable content hashes

### Anti-bloat policy

Keep the graph useful, not exhaustive.

1. Prefer `GFIL` plus selective semantic nodes over a node per AST detail.
2. Prefer document structure as facts/metadata on `GDOC`, not a node per heading by default.
3. Create `GSYM` only for public API, exported entities, and navigation-relevant definitions.
4. Store low-value details as properties or sidecar metadata when possible.
5. Skip noisy paths by default, including generated output, vendored code, caches, and lockfile internals.
6. Use configurable caps per file and per document to prevent pathological growth.
7. Promote something to a node only when it has semantic value, backlinks, or user-facing navigation value.

Suggested defaults:

- Markdown: one `GDOC` per indexed document; keep headings as compact facts unless explicit section nodes are later added.
- Code: top-level exports, classes, significant functions, and types. Store methods/fields as compact facts on the owning `GSYM` when that avoids graph bloat.
- Archives/binary/non-text files: treat as `GFIL` only unless a specialized parser exists.

## Proposed Graph Model

### Nodes

Use short, non-conflicting type codes for indexed filesystem nodes. Existing single-letter codes such as `D`, `F`, and `S` already have meanings in the current KG serializer/validator, so index-specific types should not reuse them.

Manual node types:

- `DIR` - directory added and maintained manually.
- `FIL` - file added and maintained manually.
- `DOC` - document added and maintained manually.
- `SYM` - symbol added and maintained manually.

Generated node types:

- `GDIR` - directory generated by `kg-index` from a filesystem scan.
- `GFIL` - file generated by `kg-index` from a filesystem scan.
- `GDOC` - document generated by `kg-index` from parsed file content.
- `GSYM` - symbol generated by `kg-index` from parsed code.

Optional later, only if there is a concrete navigation or query need:

- `PKG` / `GPKG`
- `MOD` / `GMOD`
- `SEC` / `GSEC`

Do not add separate default node types for class/function/method/field/import. Keep them as `SYM`/`GSYM` with a compact `kind` fact/attribute, and store methods/fields as compact facts on the owning class symbol unless they need direct backlinks.

### Generated Ownership

`G...` means the graph element is owned by the indexer, not by the user.

This applies to both nodes and edges:

- generated nodes use generated node types, e.g. `GDIR`, `GFIL`, `GDOC`, `GSYM`
- generated edges use generated relation types, e.g. `GCONTAINS`, `GHAS_DOC`, `GHAS_SYMBOL`, `GDEFINES`, `GIMPORTS`, `GREFERENCES`
- manual edges keep normal relation types, e.g. `CONTAINS`, `LINKS_TO`, `REFERENCES`, even when they point to `G...` nodes

Rules:

1. `kg-index` may create, update, replace, or delete generated nodes and generated edges during indexing.
2. Normal `kg` edit commands must reject edits/deletes of generated nodes and generated edges.
3. Manual nodes may link to `G...` nodes using manual edges.
4. Manual edges must not be rewritten into generated edges just because one endpoint is generated.
5. Generated edges may connect generated nodes to generated nodes freely when derived from the scan.
6. Generated edges may connect generated nodes to manual nodes only when the relationship is explicitly derived or configured; avoid surprising user-authored semantics.
7. User annotations about generated nodes should be represented as separate manual nodes or manual edges, not by editing generated nodes or generated edges directly.
8. Deleting a generated node during reindexing must not delete manual nodes or manual edges. It should remove only indexer-owned edges, or leave a tombstone/redirect when manual edges still reference it.

This keeps generated structure refreshable while still allowing the manually curated KG to use indexed files and symbols as stable anchors.

### Suggested meaning

- `DIR` / `GDIR` - physical folder on disk.
- `FIL` / `GFIL` - physical file on disk.
- `DOC` / `GDOC` - semantic wrapper for a human-readable file.
- `SYM` / `GSYM` - code-level entity from AST, e.g. class, function, type, constant, or exported definition.

### Identity convention

Use path-based identities for indexed code symbols so duplicates remain unambiguous.

1. `GFIL` identity is the indexed root plus repository-relative path, e.g. `repo:main/src/main/java/com/acme/App.java`.
2. `GSYM` identity is derived from the indexed root, defining file, and symbol path within that file.
3. The final segment is the symbol name; the rest captures the physical location.

Example for Java:

- `GSYM: repo:main/src/main/java/com/acme/App.java/App`
- `GSYM kind=class name=App qn=com.acme.App`
- `F k=class`
- `F m main(java.lang.String[])`
- `F f service:UserService`
- `F f port:int`

This keeps `App` as a display name, not a global identity, and avoids collisions when the same class name appears in multiple files. Path-based identity is storage identity, not the only semantic truth; keep normalized names and source spans as metadata/facts for search and refactoring support.

### Relations

Suggested relation set for indexing:

- `GCONTAINS`
- `GPART_OF`
- `GHAS_DOC`
- `GHAS_SYMBOL`
- `GDEFINES`
- `GREFERENCES`
- `GLINKS_TO`
- `GIMPORTS`
- `GEXPORTS`
- `GCALLS`
- `GIMPLEMENTS`
- `GEXTENDS`

Manual equivalents without the `G` prefix may exist for user-authored relationships, but `kg-index` must not write them. If you want to minimize schema churn, `GHAS` can be kept as a fallback for pure generated containment, but it is too coarse for code/document parsing.

## Parsing Strategy

### Document formats

1. Markdown
2. DOCX
3. plain text
4. PDF, best effort only
5. JSON/YAML/TOML/XML

### Code formats

Use `tree-sitter` for code parsing.

Why:

1. multi-language support
2. stable AST extraction
3. import/export/symbol discovery
4. fewer handwritten regex parsers

### Extraction targets

1. document titles and headings
2. code symbols and nesting
3. imports/exports
4. intra-file references and links
5. source spans
6. parse quality flags

## Storage / Incremental Behavior

1. Hash files and skip unchanged content.
2. Store scan state per root.
3. Rebuild only changed subtrees when possible.
4. Keep parser failures as first-class warnings, not silent drops.
5. Skip unknown file types when `scan_ignore_unknown` is enabled; treat them as `GFIL` only when explicitly allowed or when a parser recognizes them.
6. Track indexer-owned edges with `G...` relation types so reindexing can replace generated structure without deleting user-authored links.
7. Preserve manual edges pointing at `G...` nodes across reindexing; if the target disappears, mark the edge dangling or redirect it instead of silently dropping it.

## Schema Extensions Needed

### Node type extensions

Add canonical types for indexing so edge validation can enforce them:

- `DIR`
- `FIL`
- `DOC`
- `SYM`
- `GDIR`
- `GFIL`
- `GDOC`
- `GSYM`

### Validation updates

1. Add new types to `VALID_TYPES`.
2. Add prefix/code mappings for them.
3. Add generated edge relation types to validation, e.g. `GCONTAINS`, `GHAS_DOC`, `GHAS_SYMBOL`, `GDEFINES`, `GIMPORTS`, `GREFERENCES`.
4. Extend schema docs to describe required fields per new type.
5. Extend `EDGE_TYPE_RULES` for generated containment and symbol/document relations.
6. Reject normal edit/delete commands for `G...` nodes and `G...` edges unless the caller is `kg-index` or an explicit maintenance command.
7. Allow manual nodes and manual edges to reference `G...` nodes.

### Property extensions

Add fields for:

- `path`
- `relative_path`
- `file_hash`
- `mime_type`
- `language`
- `extension`
- `byte_size`
- `mtime`
- `start_line` / `end_line`
- `parser`
- `parse_status`
- `parse_error`

## Recommended Implementation Split

1. `kg` remains the UX entrypoint.
2. `kg-index` owns scanning and parsing.
3. Shared graph store remains authoritative.
4. Parser libraries stay isolated from the main CLI binary.

## Open Decisions

1. Should `DOC`/`GDOC` be a real node type, or should `FIL`/`GFIL` carry semantic document data?
2. What exact CLI commands are allowed to create/delete manual edges connected to `G...` nodes?
3. Should deleted/generated-renamed nodes use tombstones, redirects, or immediate removal when manual edges reference them?
4. Should code symbols be stored as separate nodes for all languages, or only for high-value languages first?
5. Should PDF/DOCX indexing be best-effort only in MVP?
6. Should the graph keep both physical containment and semantic links, or only semantic links?
